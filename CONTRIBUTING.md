# 路线数据贡献说明

欢迎补充或纠正 `wangshifu-data.json` 中的路线点。

贡献者只需提供：

- 视频链接或 BV 号；
- 任意一条你确认的信息，例如地点、当天里程、吃了什么，或者现有内容哪里不对。

不知道经纬度、准确时间点、可信度或摘要都没关系，维护者会负责查证、定位和整理成统一格式。视频时间点、路牌截图、口播等依据属于加分项，不是提交门槛。

可信度建议：

- `标题确认`：视频标题直接出现地点或里程。
- `画面确认`：路牌、码表或地图画面可以确认。
- `口播确认`：视频口播明确说明。
- `路线推断`：仅能根据前后城市推断，必须明确标注。

不要把两点之间的直线距离当作真实骑行里程，也不要把公开视频落脚点描述成 GPS 实时位置。

最省事的方式是使用网页右上角的“补路线点”，写一句你发现的内容后复制提交。熟悉 GitHub 的贡献者也可以直接修改 JSON 或提交 Pull Request。

## 点赞采纳

- 每条建议作为一个 GitHub Issue，观众用 👍 reaction 投票。
- 同一期视频出现多条建议时，优先核验点赞较高的内容。
- 达到 3 个 👍 的建议会自动标记为 `票选-优先`。
- 点赞代表核验优先级，不替代事实核查；维护者确认视频内容后再写入正式数据。

## 待核验如何通过

采用四级状态，尽量让维护者只做最后判断：

1. `机器候选`：公开评论含地点、里程或饮食关键词，并达到 10 赞；自动进入候选池。
2. `票选优先`：GitHub 建议达到 3 个 👍；自动排到核验队列前面。
3. `证据充分`：满足下面任意一项，由维护者确认：
   - 视频标题、口播、字幕、路牌或码表画面直接证明，并附视频时间点；
   - UP 主本人回复确认；
   - 两条相互独立的公开信息一致，其中至少一条能对应视频内容。
4. `已采纳`：地点同时完成坐标核对，里程没有被误当作直线距离；写入正式数据并保留证据链接。

点赞只决定“先核验谁”，不能单独证明事实。单条评论、相邻路线推断和 AI 识别结果都不会自动覆盖正式地图；若证据冲突，继续保留在待核验区。

## 维护者采纳操作

只有仓库维护者账号 `tlxiansen` 可以让“已采纳”生效：

1. 打开仓库的 `Issues` 页面并进入一条待核验建议。
2. 核对视频、时间点和建议内容。
3. 在右侧 `Labels` 中选择绿色的 `已采纳`。
4. 系统自动把可识别的地点、里程、美食、住宿或关键时间点写入 `wangshifu-data.json`。
5. 写入成功后，Issue 会出现数据提交链接并自动关闭；网站通常几分钟后更新。

其他账号添加“已采纳”会被自动移除。无法识别 BV 号或没有对应视频数据时，系统不会修改正式路线，并会在 Issue 中说明原因。

## B站评论自动转为 Issue

- 每次数据更新会扫描合集内每期视频的公开热门评论。
- 只保留与地点、路线、里程、住宿、美食、预算等相关且至少 10 赞的评论。
- 新候选会自动创建带 `route-data`、`待核验`、`auto-bilibili` 标签的 GitHub Issue。
- 评论 ID 会写入隐藏标记用于去重，已经创建或关闭的建议不会重复生成。
- 为避免短时间刷屏，每次运行最多创建 10 条；剩余候选会在后续运行继续创建。
- 自动生成只代表“值得核验”，不会直接修改正式路线；仍需 `tlxiansen` 核对后添加 `已采纳`。
